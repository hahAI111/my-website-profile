# Troubleshooting — Issues Encountered During Development & Their Solutions

This document records all the important problems encountered during this project's journey from scratch to deployment, along with their solutions.

---

## Issue #1: SQLite → PostgreSQL Syntax Incompatibility

### Problem
The project was initially developed using SQLite, then migrated to Azure PostgreSQL. After directly swapping the connection code, all SQL statements threw errors.

### Root Cause
| | SQLite | PostgreSQL (psycopg2) |
|---|--------|----------------------|
| Placeholder | `?` | `%s` |
| Execution | `conn.execute(sql)` | `cur = conn.cursor(); cur.execute(sql)` |
| Auto-increment ID | `cur.lastrowid` | `INSERT ... RETURNING id` + `cur.fetchone()[0]` |
| Dict rows | `conn.row_factory = sqlite3.Row` | `cursor_factory=psycopg2.extras.RealDictCursor` |

### Solution
Modified SQL syntax in every route function:
- `?` → `%s`
- `conn.execute()` → `cur = conn.cursor()` + `cur.execute()` + `cur.close()`
- `cur.lastrowid` → `RETURNING id` + `cur.fetchone()[0]`
- Admin stats uses `RealDictCursor` for dictionary-format results

## Issue #2: Azure Connection String Format Mismatch → 504 Gateway Timeout

### Problem
After pushing code to Azure, the website returned **504 Gateway Timeout**.

### Root Cause
The PostgreSQL Connection String auto-generated by Azure "Web App + Database" is in **ADO.NET format** (designed for C#/.NET):
```
Database=aimeelan-database;Server=aimeelan-server.postgres.database.azure.com;User Id=prjxaadsjr;Password=xxx
```

But Python's psycopg2 requires **libpq format**:
```
host=aimeelan-server.postgres.database.azure.com dbname=aimeelan-database user=prjxaadsjr password=xxx
```

Similarly, the Azure Redis Connection String format is also incompatible:
```
# Format provided by Azure
aimee-cache.redis.cache.windows.net:6380,password=xxx,ssl=True,abortConnect=False

# Format required by Python redis-py
rediss://:xxx@aimee-cache.redis.cache.windows.net:6380/0
```

The gunicorn worker failed to connect to the database on startup → worker startup timeout → Azure front-end proxy returned 504.

### Solution
Wrote two parsing functions to automatically convert the format:

```python
def _parse_pg_conn(raw):
    """Convert ADO.NET format to libpq format."""
    if not raw or raw.startswith("host="):
        return raw
    parts = dict(p.split("=", 1) for p in raw.split(";") if "=" in p)
    return f"host={parts['Server']} dbname={parts['Database']} user={parts['User Id']} password={parts['Password']}"

def _parse_redis_conn(raw):
    """Convert Azure Redis format to rediss:// URL."""
    if not raw or raw.startswith("redis"):
        return raw
    parts = dict(p.split("=", 1) for p in raw.split(",") if "=" in p)
    host_port = raw.split(",")[0]
    password = parts.get("password", "")
    use_ssl = parts.get("ssl", "False").lower() == "true"
    scheme = "rediss" if use_ssl else "redis"
    return f"{scheme}://:{password}@{host_port}/0"
```

### Lesson Learned
Azure's "Web App + Database" template targets the .NET ecosystem — the generated connection strings are not Python-friendly. When using non-.NET languages, manual parsing is required.

## Issue #3: Azure Connection String Environment Variable Prefix

### Problem
The code uses `os.environ.get("AZURE_POSTGRESQL_CONNECTIONSTRING")` to read the connection string, but the value was empty on Azure.

### Root Cause
Azure App Service's **Connection Strings** (configured under Environment variables → Connection strings tab) automatically add a prefix when injected as environment variables:

| Connection String Type | Environment Variable Prefix |
|----------------------|------------|
| Custom | `CUSTOMCONNSTR_` |
| SQL Server | `SQLCONNSTR_` |
| SQL Azure | `SQLAZURECONNSTR_` |
| MySQL | `MYSQLCONNSTR_` |
| Redis Cache | `REDISCACHECONNSTR_` |
| PostgreSQL | `POSTGRESQLCONNSTR_` |

So `AZURE_POSTGRESQL_CONNECTIONSTRING` (Custom type) became:
```
CUSTOMCONNSTR_AZURE_POSTGRESQL_CONNECTIONSTRING
```

And the Redis name `azure_redis_cache` (Redis Cache type) became:
```
REDISCACHECONNSTR_azure_redis_cache
```

### Solution
The code checks both name variants:
```python
_raw_pg = (
    os.environ.get("AZURE_POSTGRESQL_CONNECTIONSTRING")
    or os.environ.get("CUSTOMCONNSTR_AZURE_POSTGRESQL_CONNECTIONSTRING")
    or "host=localhost ..."
)
```

### Lesson Learned
Azure **App Settings** vs **Connection Strings** are two different things:
- **App Settings** → Become environment variables directly, names unchanged
- **Connection Strings** → Get a prefix prepended when becoming environment variables

It's recommended to use App Settings for storage — the code is simpler that way. However, Azure's "Web App + Database" auto-creates entries under Connection Strings, so we need to handle both.

## Issue #4: Azure PostgreSQL Requires SSL Connection

### Problem
Even after fixing the connection string format, psycopg2 still failed to connect to Azure PostgreSQL.

### Root Cause
Azure PostgreSQL Flexible Server enforces SSL connections by default. Local psycopg2 does not use SSL by default.

### Solution
Added `sslmode="require"` when connecting:
```python
def get_db():
    conn = psycopg2.connect(DATABASE_URL, sslmode="require")
    return conn
```

## Issue #5: GitHub Push Rejected — Remote Contains Work

### Problem
The first `git push` was rejected:
```
! [rejected] main -> main (fetch first)
error: failed to push some refs
```

### Root Cause
When configuring GitHub Actions deployment through the Azure portal, Azure automatically pushed the following to the GitHub repo:
- `.github/workflows/main_aimeelan.yml` (CI/CD workflow file)
- `LICENSE`

The local repository was a fresh `git init` with no common commit history with the remote.

### Solution
```bash
git pull origin main --allow-unrelated-histories
# Merges two unrelated histories
git push
```

`--allow-unrelated-histories` tells git: "I know these two branches have no common ancestor — force the merge."

## Issue #6: PowerShell's curl Is Not Real curl

### Problem
Testing the website with the `curl` command threw an error:
```
Missing an argument for parameter 'SessionVariable'
```

### Root Cause
In Windows PowerShell, `curl` is an alias for `Invoke-WebRequest`, not Linux's curl. The parameter syntax is completely different.

### Solution
Use PowerShell's native syntax instead:
```powershell
Invoke-WebRequest -Uri "https://aimeelan.azurewebsites.net" -UseBasicParsing
```

## Issue #7: Gmail App Password Configuration

### Problem
Needed to send email notifications via Gmail SMTP, but Google doesn't allow using your regular Gmail password.

### Root Cause
Google requires you to enable 2-Step Verification first, then use an **App Password** (a 16-character dedicated password) instead of your account password.

### Solution Steps
1. Google Account → Security → Enable **2-Step Verification**
2. Visit https://myaccount.google.com/apppasswords
3. Enter an app name → Generate → Receive a 16-character password
4. Set `SMTP_PASS` = that 16-character password in Azure App Settings

**Important Notes:**
- The App Password is shown only once — save it immediately
- Never hardcode the App Password in source code; only store it in Azure environment variables
- If 2FA is not enabled, the App Passwords page won't appear

If the website has issues again, check in the following order:

| Step | Check | Command / Location |
|------|-------|--------------------|
| 1 | GitHub Actions succeeded? | https://github.com/hahAI111/aimeewebpage/actions |

| 4 | Environment variables complete? | Environment variables → App settings + Connection strings |
| 5 | Check application logs | `az webapp log tail --name aimeelan --resource-group aimee-test-env` |
| 6 | PostgreSQL reachable? | VNet Integration + Private Endpoint status |
